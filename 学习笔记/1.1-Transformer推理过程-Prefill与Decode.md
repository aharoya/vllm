# 1.1 Transformer 推理过程：Prefill 与 Decode

大模型生成回复分为两个阶段：**Prefill（预填充）** 和 **Decode（解码）**。理解这两个阶段的区别，是理解 vLLM 一切调度策略与性能优化的基石。

---

## 一、整体流程概览

用户问 "1+1等于几？"，模型回答 "等于2"。过程如下：

```
用户输入: "1+1等于几？"
            │
            ▼
┌──────────────────────────┐
│  Prefill 阶段（1 次）     │  ← 并行处理 prompt 所有 token
│  输入：全部 prompt token  │
│  输出：第 1 个新 token    │
└──────────────────────────┘
            │
            ▼  第 1 个字: "等"
┌──────────────────────────┐
│  Decode 阶段（循环 N-1 次）│  ← 每次只处理 1 个新 token
│  输入：上一步生成的 token  │
│  输出：下一个 token       │
└──────────────────────────┘
            │
            ▼  第 2 个字: "于", 第 3 个: "2", 直到 <结束符>
```

生成 3 个字的回复 = 1 次 Prefill + 2 次 Decode。

---

## 二、Prefill 阶段

### 做了什么

把用户的 prompt 全部 token **一次性、并行地**喂给模型，经过所有 Transformer 层，产生第一个输出 token。

### 详细步骤

```
Prompt: "1+1等于几？"  (7 个 token)

步骤 1: 7 个 token 同时进入第 1 层 Transformer
        每个 token 都算出自己的 Q、K、V
        Attention: 每个 token 看它前面的所有 token
        输出: 7 个融合了上下文的向量

步骤 2: 这 7 个向量同时进入第 2 层 Transformer
        同样的过程重复一遍
        ...

步骤 N: 经过全部 32 层（假设 Llama-7B）
        最后一层的最后一个位置的输出
        → LM Head → logits → argmax → 第 1 个生成 token
```

### 关键特征

**1. 并行处理**

7 个 token 在每一层都是同时计算的，不需要等前一个算完。这是因为 prompt 是完整给定的，模型知道所有 token 的内容。

```
"1" 算 Attention 时可以看到     "1"
"+" 算 Attention 时可以看到     "1" "+"
"1" 算 Attention 时可以看到     "1" "+" "1"
"等" 算 Attention 时可以看到    "1" "+" "1" "等"
"于" 算 Attention 时可以看到    "1" "+" "1" "等" "于"
"几" 算 Attention 时可以看到    "1" "+" "1" "等" "于" "几"
"？" 算 Attention 时可以看到    "1" "+" "1" "等" "于" "几" "？"
                            ↑ 全部同时算完
```

**2. 一次性生成全部 KV Cache**

每个 token 经过每一层时，它的 K 和 V 都被算出来并存起来（因为后续 Decode 阶段要用）。这些 KV Cache 构成了解码阶段不再重算的基础。

**3. 计算量集中**

如果 prompt 有 4096 个 token，那就是 4096 个 token 同时过全部 32 层，矩阵非常大。GPU 的算力被充分利用。

**4. 瓶颈：Compute-bound（计算受限）**

GPU 的计算单元忙不过来，显存带宽反而不是瓶颈。提升 Prefill 速度的主要手段是加算力或增大矩阵并行度。

### 对用户的影响

Prefill 的耗时决定了 **TTFT（Time To First Token，首 token 延迟）**——用户从发送请求到看到第一个字的时间。

```
TTFT = Prefill 耗时 + 少量系统开销
```

Prompt 越长，Prefill 越慢，TTFT 越大。如果你给模型发了一篇 5000 字的长文，Prefill 可能就要几百毫秒。

vLLM 中缓解长 prompt 的手段：**Chunked Prefill**——把长 prompt 切成多个小块，分多次 Prefill，避免一次 Prefill 阻塞所有请求的 Decode。

---

## 三、Decode 阶段

### 做了什么

拿到上一个 token 之后，**只把这一个 token 喂给模型**，经过所有层，输出下一个 token。然后把这个新 token 再喂进去，如此循环。

### 详细步骤

```
已经生成 "等"，接下来要生成 "于"

步骤 1: 只输入 token "等"（1 个 token）
        Embedding → 进入第 1 层

步骤 2: 第 1 层 Attention
        Q = "等" 的 Query（1 个，新算的）
        K = [缓存中 7 个历史 Key] + [新算的 1 个 Key]   ← 从缓存读
        V = [缓存中 7 个历史 Value] + [新算的 1 个 Value] ← 从缓存读
        Attention(Q, K, V) → 融合了前面 7 个 token 信息的新向量

步骤 3: 这个向量继续过第 2 层、第 3 层...第 32 层
        每层都做同样的 KV 缓存拼接

步骤 4: 最后一层输出 → LM Head → logits → argmax → 下一个 token "于"

然后把 "于" 再喂进去 → 得到 "2" → 把 "2" 喂进去 → 得到 <结束符> → 停止
```

### 关键特征

**1. 串行、无法并行**

必须先算出 "于"，才能把 "于" 喂进去算出下一个。所以 Decode 没办法并行。

**2. 每次只算 1 个 token 的新 KV**

每次只产生 1 个 token 的 1 组 K、V，拼到历史 KV Cache 后面。历史的不重算。

```
第 1 次 Decode: 算 1 组 KV, 缓存有 7+1=8 组
第 2 次 Decode: 算 1 组 KV, 缓存有 8+1=9 组
...
第 100 次 Decode: 算 1 组 KV, 缓存有 7+100=107 组
```

每次计算量是恒定的（只算 1 个 token），但每次要读取的缓存越来越大——要从显存里读 107 组 KV 到计算单元。

**3. 瓶颈：Memory-bound（显存带宽受限）**

Decode 阶段 GPU 的大部分时间不是在计算，而是在**等数据从显存搬到计算单元**。

- 计算一个 token 的 Q 只需要 1 组 embedding
- 但 Attention 需要读取全部历史 KV（可能有几千组）
- 读取几千组数据的时间远大于计算这 1 组的时间

> 类比：你在图书馆写论文。Prefill 是一次查 100 本书，然后埋头写 10 页（计算是瓶颈）。Decode 是每次查 1 本书写 1 行，但你要先跑遍整个图书馆把 100 本书搬到桌上（搬书是瓶颈）。

**4. 对 batch 的依赖**

如果一个请求单独做 Decode，GPU 的计算单元大部分时间闲着（等数据搬过来）。但如果同时有 100 个请求都在做 Decode，搬一批数据可以同时给 100 个请求算，GPU 利用率就上来了。

### 对用户的影响

Decode 的耗时决定了 **TPOT（Time Per Output Token，每 token 延迟）**——生成过程中每个字之间的间隔。

```
TPOT = 单次 Decode 耗时
```

用户感受到的"出字速度" = 1 / TPOT。TPOT 越小，字蹦得越快。

## 四、Prefill vs Decode 对比

```
                         Prefill                    Decode
─────────────────────────────────────────────────────────────
输入内容            全部 prompt token          上一步生成的 1 个 token
输入长度            可长可短（几十到几万）    固定为 1
执行次数            每个请求 1 次              每个输出 token 1 次
能否并行            是（prompt token 之间）   否（必须等上一步结果）
每次新算 KV 组数    prompt 长度那么多         1 组
读取 KV Cache       无（首次填充）            读全部历史
瓶颈类型            Compute-bound             Memory-bound
GPU 算力利用率      高（大矩阵运算）          低（大部分时间在等数据）
影响什么延迟        TTFT（首 token 延迟）     TPOT（每 token 延迟）
─────────────────────────────────────────────────────────────
```

### 一张图理解

```
时间轴 ──────────────────────────────────────────────────→

Prefill                              Decode
┌─────────────┐    ┌──┐ ┌──┐ ┌──┐ ┌──┐ ┌──┐
│ prompt      │    │新 │ │新 │ │新 │ │新 │ │新 │
│ 全部 token  │ →  │  │→│  │→│  │→│  │→│  │→ ...
│ 并行处理    │    │  │ │  │ │  │ │  │ │  │
│             │    │  │ │  │ │  │ │  │ │  │
│ 算力瓶颈    │    │  带宽瓶颈                     │
└─────────────┘    └──┘ └──┘ └──┘ └──┘ └──┘
     ↑               ↑               ↑
   TTFT          TPOT ①          TPOT ②
```

## 五、为什么 vLLM 关心这个区别

两种阶段的瓶颈截然不同，所以 vLLM 的调度策略必须区别对待：

### Chunked Prefill（分块预填充）

长 prompt 一次性 Prefill 会霸占 GPU 很久，导致正在 Decode 的请求被卡住（用户看到字不蹦了，体验很差）。

```
方案：把 8192 token 的 prompt 切成 4 块，每块 2048 token
     每次只 Prefill 一块，穿插 Decode 步骤

时间线：
  [prefill 2048] → [decode 一批请求] → [prefill 2048] → [decode] → ...
                           ↑ 不卡了
```

### 混合调度

vLLM 的 Scheduler 每一步决定：多少请求做 Prefill、多少请求做 Decode。需要平衡：

- 如果只做 Prefill → Decode 请求超时（卡住）
- 如果只做 Decode → 新请求永远不开始（饿死）
- Prefill 太多 → GPU 算力不够，大家都慢
- Decode 太多 → 显存带宽打满，同样慢

### 异构分离部署（Disaggregated Serving）

一个极端的优化方案是：prefill 用一台机器（要算力），decode 用另一台（要带宽），中间通过 KV Cache 传输连接。

```
用户 → Prefill 节点（算力强） → KV Cache 传输 → Decode 节点（带宽大） → 输出
```

---

## 六、总结

1. **Prefill = 一次性并行处理 prompt，算力瓶颈**，决定了第一个字多快出来（TTFT）
2. **Decode = 每次 1 个 token 串行生成，带宽瓶颈**，决定了字和字之间的间隔（TPOT）
3. vLLM 的调度器根据这两个阶段的不同特点制定不同的调度策略
4. 理解了 Prefill/Decode，就理解了后面要学的 Chunked Prefill、Continuous Batching、投机解码等优化的动机
