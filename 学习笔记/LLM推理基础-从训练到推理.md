# LLM 推理基础：从训练到推理

本文梳理大语言模型（LLM）从训练到推理的完整链路，并解释 vLLM 在这一链路中的位置与价值。

配套代码：`学习笔记/demo_qkv_inference.py`（可直接运行）。

---

## 1. 大语言模型是什么

大语言模型本质上是一个**数学函数**：

```
输入文字 → 模型 → 输出文字

"北京有哪些景点" → [神经网络计算] → "北京有故宫、长城、天坛..."
```

这个函数的参数（权重）是通过大量文本训练得到的。训练完成后，参数就固定下来，不再改变。

---

## 2. 完整链路：从文字到预测

一个文字请求经过以下几步变成模型的输出：

```
文字 "我爱北京"
   ↓  ① 分词（Tokenization）
token IDs  [1053, 2847, 12648]
   ↓  ② 嵌入（Embedding）
向量  [v₀, v₁, v₂]  (每个是 hidden_dim 维浮点数)
   ↓  ③ 线性变换
Q = v × W_Q     K = v × W_K     V = v × W_V
   ↓  ④ Attention
输出 = softmax(Q·K^T/√d) × V
   ↓  ⑤ 输出投影（LM Head）
logits → argmax → 下一个 token ID → 查表 → 文字
```

### 2.1 分词（Tokenization）

模型不认识汉字，只认识数字。分词器把文字切成"词块"，每个词块对应一个整数 ID。

```
输入："我爱北京"
  → ["我", "爱", "北京"]    每个元素叫一个 token
  → 查词汇表
  → [1053, 2847, 12648]    三个整数
```

### 2.2 嵌入（Embedding）

整数本身没有语义信息。Embedding 层是一个大表格（行数 = 词汇表大小，列数 = hidden_dim），把每个 token ID 映射为一个浮点数向量。

```
1053 → [0.12, -0.34, 0.78, ...]    (长度 = hidden_dim，如 4096)
2847 → [0.45,  0.21, -0.11, ...]
12648 → [-0.03, 0.67, 0.33, ...]
```

语义相近的词，学出来的向量距离也近（"苹果"和"香蕉"的向量在高维空间中靠得更近）。

### 2.3 Q、K、V 从哪里来

每个 token 的 embedding 向量，分别乘三个不同的矩阵 W_Q、W_K、W_V，得到 Q、K、V。

```
         token "北京" 的 embedding
                  │
        ┌─────────┼─────────┐
        ↓         ↓         ↓
     × W_Q      × W_K     × W_V
        ↓         ↓         ↓
       Q          K         V
    (Query)    (Key)    (Value)
```

**为什么同一个向量要乘三个不同的矩阵？**

| 角色 | 来源 | 作用 | 类比 |
|---|---|---|---|
| **Q**（Query） | embedding × W_Q | "我想从其他词那里了解什么？" | 你在图书馆**提出的问题** |
| **K**（Key） | embedding × W_K | "我能被哪些词匹配到？" | 每本书的**标题** |
| **V**（Value） | embedding × W_V | "我的实际内容是什么？" | 每本书的**正文** |

W_Q、W_K、W_V 在训练中学出来，训练完成后就冻结不变。

### 2.4 Attention 计算

Attention 的核心思想：**让每个 token 看看它前面的所有 token，决定自己应该从每个位置吸收多少信息。**

```
步骤：
① Q 和所有 K 做点积 → 得到注意力分数
   scores = Q × K^T / √d     (除以 √d 防止数值过大)

② causal mask：遮掉"未来"的 token
   只能看自己和自己前面的，不能偷看后面的

③ softmax：把分数变成权重（加起来等于 1）
   分数大的 → 权重高 → 这个位置的信息被吸收得多

④ 用权重对 V 加权求和 → 融合了上下文信息的输出
   output = 0.1×V₀ + 0.2×V₁ + 0.6×V₂ + ...
```

## 3. 训练：W_Q、W_K、W_V 是怎么来的

### 3.1 训练的核心循环

```
for 每一批数据:
    ① 输入一句话（整句一次性给）
    ② 前向传播（forward）：算 QKV → Attention → 预测下一个词
    ③ 计算损失（loss）：预测 vs 正确答案，差距多大？
    ④ 反向传播（backward）：从 loss 往回算梯度
    ⑤ 更新参数：按梯度微调 W_Q、W_K、W_V 等所有参数
    ⑥ 重复几万亿次 → 参数收敛 → 保存模型文件
```

### 3.2 训练与推理的关键区别

| | 训练 | 推理 |
|---|---|---|
| 输入方式 | 整句话一次性给 | 逐 token 生成 |
| 前向传播 | 有 | 有 |
| 反向传播 | 有（调参数） | **无**（参数冻结） |
| W_Q/W_K/W_V | 一直在变 | 永远不变 |
| QKV 计算 | 一次性算所有 token | 每步都要算新 token 的 QKV |

## 4. 推理：为什么需要 KV Cache

### 4.1 逐 token 生成的问题

推理是自回归的：每生成一个 token，拼到输入末尾，再生成下一个。

```
第 1 步：输入 "中国的首都是"         → 算 6 组 QKV → 输出 "北"
第 2 步：输入 "中国的首都是北"       → 算 7 组 QKV → 输出 "京"
第 3 步：输入 "中国的首都是北京"     → 算 8 组 QKV → 输出 "。"
...
第 100 步：输入 "中国的首都是北京..." → 算 100+ 组 QKV
```

观察到什么？**前几步算过的 QKV，后面又在重复算。** 生成 N 个 token，总 KV 计算量是 O(N²)。

### 4.2 KV Cache 的解决方案

**K 和 V 一旦算出来就不变了，存起来下次直接用。Q 每次都是新的，必须重算。**

```
第 1 步：输入 "中国的首都是" → 算 6 组 KV，全存入缓存 → 输出 "北"
第 2 步：只输入 "北" → 算 1 组 KV → 拼上缓存里的 6 组 → Attention 看到 7 个 token → 输出 "京"
第 3 步：只输入 "京" → 算 1 组 KV → 拼上缓存里的 7 组 → Attention 看到 8 个 token → 输出 "。"
```

**传给 model 的输入从"全部历史 token"变成了"1 个新 token"，但 Attention 内部通过缓存拼接，看到的上下文和传全部 token 时完全一样。**

### 4.3 对比

| | 无 KV Cache | 有 KV Cache |
|---|---|---|
| 每次传给 model 的 token 数 | 3 → 4 → 5 → 6 → ... | 3 → **1** → **1** → **1** → ... |
| 每次新算的 KV 组数 | 3 → 4 → 5 → 6 → ... | 3 → **1** → **1** → **1** → ... |
| Attention 看到的 token 数 | 3 → 4 → 5 → 6 → ... | 3 → 4 → 5 → 6 → ...（来自缓存拼接） |
| 生成 N 个 token 的计算量 | O(N²) | O(N) |

## 5. vLLM 的位置

```
┌─────────────────────────────────────────────────────────┐
│                  vLLM 的战场                             │
│                                                         │
│  KV Cache 的管理与优化：                                  │
│   • PagedAttention — 像 OS 虚拟内存一样管理 KV 缓存块     │
│   • Continuous Batching — 动态加入/移除请求               │
│   • Prefix Caching — 相同前缀共享缓存                     │
│   • Chunked Prefill — 大 prompt 分块，避免阻塞 decode     │
│                                                         │
│  目标：用有限的 GPU 显存，同时服务尽可能多的请求，且尽量快  │
└─────────────────────────────────────────────────────────┘
```

vLLM 不改变模型的计算逻辑——Attention 还是那个 Attention，QKV 还是那个 QKV。vLLM 做的是**在"怎么存、怎么调度、怎么并行"这一层做到极致**。
